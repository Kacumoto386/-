# 课程预约功能设计方案

## 需求分析

现有的"上课记录"是所有课程行为的终点（签到、完成），但缺少独立的"预约—签到—完成"流程。

## 设计思路

**不新增Sheet**，而是在现有的 `上课记录` 表基础上，利用已有的 `上课状态`（已预约/已签到/已完成/已取消/爽约）和 `预约日期` 字段，增加独立的预约UI和业务逻辑。

### 需要改造的范围

| 模块 | 改动 | 改动量 |
|------|------|:------:|
| `core/business.py` | 新增预约相关方法 | 中 |
| `gui/booking_frame.py` | **新增**独立预约管理界面 | 大 |
| `gui/main_window.py` | 导航栏增加"预约管理"入口 | 小 |
| `gui/class_frame.py` | 上课记录改为从预约签到进入 | 中 |

## 详细方案（推荐）

### 方法A：独立预约Sheet（推荐✅）

新增 Sheet `预约管理`，独立存储预约数据，与上课记录分离。

**预约管理表头**：
```
预约编号 | 预约日期 | 预约时间段 | 会员编号 | 会员姓名 | 课程编号 | 课程名称 | 教练编号 | 教练姓名 | 预约状态 | 签到时间 | 消耗课时 | 备注
```

**预约状态**：已预约 → 已签到 → 已完成 / 已取消

**优势**：
- 预约和上课记录功能分离，不互相影响
- 可以看到所有已预约但未上课的记录
- 支持教练排班视图

**缺点**：
- 需要新增Sheet和完整的CRUD
- 上课时需要从预约签到，流程变化

### 方法B：复用上课记录表（轻量级✅）

利用已有的 `上课记录` 表，但增加独立的预约UI。预约时写入上课状态="已预约"，上课签到后改为"已签到"。

**优势**：
- 不破坏现有数据结构
- 上课记录表已有预约相关字段
- 改动最小

**缺点**：
- 预约和完成上课混在同一张表
- 过滤逻辑稍复杂

## 推荐方案：方法A（独立预约Sheet）

### Sheet设计

**预约管理** (`booking_info`) 表头：
```
预约编号 | 预约日期 | 开始时间 | 结束时间 | 会员编号 | 会员姓名 | 课程编号 | 课程名称 | 教练编号 | 教练姓名 | 预约状态 | 预约人数 | 已签到人数 | 创建时间 | 备注
```

### 业务逻辑（`core/business.py`）

```python
def add_booking(self, data)       # 创建预约
def cancel_booking(self, row)     # 取消预约
def sign_in_booking(self, row)    # 签到（预约→上课）
def get_bookings_by_date(self, d) # 按日期查预约
def get_bookings_by_member(self)  # 按会员查预约
def check_booking_conflict(self)  # 检查时间冲突
```

### GUI新增

**`gui/booking_frame.py`** - 独立的预约管理界面：
- 日历/日期选择视图
- 按日期查看全部预约
- 教练排班时间线
- 会员端: 个人预约列表
- 新增预约弹窗（选日期/时段/教练/课程）
- 签到按钮（预约→上课）
- 取消预约按钮

### 上课记录的改动

已有的上课记录界面保持不变，但增加一个"从预约签到"入口：
- 在预约界面点"签到"→ 自动填写上课记录部分字段
- 上课记录中的"预约日期"由系统自动填入

### 数据流转

```
会员预约 → 预约表(状态=已预约) → 签到 → 上课记录表(状态=已签到) + 预约表(状态=已签到)
                                     → 完成 → 上课记录表(状态=已完成) + 消耗课时
                                 取消 → 预约表(状态=已取消)
```

### 冲突检查

- 同一教练在同一时间段只能有一个预约（私教课）
- 同一会员在同一时间段只能有一个预约
- 团课/小班课检查预约人数不超过最大预约人数
